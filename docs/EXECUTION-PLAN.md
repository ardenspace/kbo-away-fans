# KBO 원정러 — V1 실행계획

> pslog-planning 5렌즈 산출물. 입력: `docs/CONCEPT.md`.
> 상류 lock-in 결정(E#)은 §5. PLAN.md task 가 `(E#)` 로 백링크한다.

## 1. 한 줄 요약

응원팀을 정하면 원정 경기 일정이 뜨고, 그 구장의 **맛집·가이드·우천 플랜B**를 한 화면에서 보고, 방문하면 **GPS 스탬프**가 찍히는 원정 팬용 앱. (Flutter + Supabase + 맥미니 일정 크롤러)

## 2. 풀려는 문제 (누구 / 왜)

- **누구**: 원정 가는 KBO 팬 중 **도시여행형** — 원정지를 통째로 즐기고 싶은 사람. (본인이 그 유저 = dogfood)
- **왜**: 낯선 도시 구장에 가면 맛집·주차·동선·우천 시 대안 정보가 블로그/카페/유튜브에 **흩어져 있어** 매번 검색 노가다. KBO 일정은 공식 오픈API가 없고 우천으로 자주 바뀜.
- **성격**: 즐기면서 만드는 / pslog dogfooding 프로젝트. 수익화 비핵심 → **과엔지니어링 금지, "내가 쓸 만큼"이 완성도 기준**.

## 3. 무엇이 다른가 (차별점)

- **10구단 원정 정보의 품질·집약**이 본체. 네이버 지도/일반 여행앱은 "야구 원정" 맥락을 모르고, 구단 공식앱은 여행/맛집을 안 다룸 — 그 사이 빈칸을 메움.
- **게임화 애니(스탬프·경로·테마컬러)는 부차 매력** — 본질은 정보 품질.

## 4. 범위 (YAGNI)

### ✅ 만든다 (V1)
1. 응원팀 설정 → 원정 경기 일정 연동
2. 구장별 가이드 (주차·좌석뷰·동선·편의점)
3. 맛집 추천 (**수작업 시딩**, 출처 링크, 읽기 전용)
4. 우천 플랜B (구장 반경 실내 놀거리)
5. 스탬프 (**GPS 근접** 발급)
6. 로그인 (**이메일 + 카카오**) + 유저 상태 클라우드 동기화
7. 네이버맵 위치 표시 + **경로 애니** (방문 구장 잇기)
8. 팀별 테마 컬러 전환

### ⏸ 안 만든다 (V1.5 / V2)
- 유저 제보 + 어드민 승인 흐름
- 반자동 크롤링 **LLM 추출 파이프라인** ("선수+장소" 멘션 추출) — LLM 선택(로컬 vs 호스티드)은 이때 결정
- 팬 후기/사진 커뮤니티
- 10구장 완주 뱃지 등 심화 게임화
- 구글 / 애플 로그인
- 분 단위 실시간 우천 푸시

## 5. 핵심 설계 결정 ★ (상류 lock-in)

| id | 결정 | 이유 / 근거 렌즈 |
|---|---|---|
| **E1** | **백엔드 = Supabase 직접 호출** (별도 API 서버 없음), **맥미니 셀프호스팅**(Docker: Postgres+Auth+Storage+PostgREST). 콘텐츠 테이블 public read, 유저 테이블 owner-scoped **RLS**, **크롤러만 service-role writer**. 앱은 Supabase SDK 그대로 — 호스티드↔셀프호스팅은 env 변수 한 줄 차이라 비종속. | 1인 dogfood·읽기 위주 → API 레이어 과함. 무료티어 7일 미사용 시 잠자기 + 비용 우려 → 셀프호스팅으로 잠자기·한도·비용 0. 맥미니 인프라+도메인 보유. (L3·승인후 재조정) |
| **E2** | **인증 = Supabase Auth**, v1 **이메일 + 카카오**만. 구글/애플은 후속. | 4종 동시 셋업(특히 애플 유료계정·심사)은 과엔지니어링(L1-4)과 충돌. (L4-2) |
| **E3** | **유저 상태(응원팀·스탬프) 클라우드 저장** — `profiles`, `stamps` + RLS. | 재설치·기기변경에도 스탬프 생존. (L3 수정·L4) |
| **E4** | **일정 = 맥미니 크롤러**(Python + cron), **2단 주기**: 평소 1일 1회 + **경기일 첫 경기 30분 전 ~ 마지막 경기 종료 1h 후 = 1분 주기**. `game_id` upsert. ➕ 앱이 경기 화면 열 때 **on-demand 상태 1회 조회**. | 공식 API 부재. 하루 최대 5경기가 *한 페이지*에 다 나옴 → 크롤 1회=요청 1개라 1분 주기여도 분당 1요청뿐, 차단 무관. on-demand로 체감 신선도↑. (L4-3 재조정) |
| **E5** | **크롤러 graceful + 텔레그램 실패 알림**. 크롤 실패해도 앱은 캐시된 DB 데이터로 동작. | HTML 변경/차단 시 앱 멈춤 방지, 1인 운영 가시성. (L3 위험·L4) |
| **E6** | **지도 = 네이버맵**(`flutter_naver_map`). **경로 애니(PathOverlay) v1 포함**. | 네이버맵 Flutter SDK가 더 성숙 + 경로 오버레이 네이티브 → "애니 많이" 컨셉에 유리. (L4-5 재조정) |
| **E7** | **스탬프 = GPS 근접 발급** — 구장 좌표 반경 임계값(기본 500m) 안에서만, 중복 방지. | 수집의 재미(아무데서나 못 찍음). 스푸핑은 dogfood라 무시. (L4-4) |
| **E8** | **맛집·플랜B·구장가이드 = 수작업 시딩, 10구장 전부**. 출처 URL 필수. **`pick_type`(선수/팬/에디터)** 필드로 목록·지도엔 「⚾선수 PICK!/🙌팬 PICK!」 배지만, 출처 링크는 **상세보기에서만** 노출. | 정보 집약이 본체(L1). 모음집 활용해 노동 감당. 배지가 v2(LLM=선수픽, 제보=팬픽)로 이어짐. (L2·L4-1·재조정) |
| **E9** | **앱 = Flutter**, 팀별 **테마 컬러 전환**. | 자체 렌더링으로 애니 자유도. (CONCEPT·L1) |
| **E10** | **키 격리** — service-role 키는 맥미니 크롤러에만, 앱은 anon key. 앱은 콘텐츠 테이블 쓰기 권한 없음. | 권한 누설 방지. (L3 위험) |

## 6. 리스크 / 영향 (실패 클래스 5버킷 + 도메인)

| 버킷 | 심문 결과 / 처리 |
|---|---|
| **도메인 특수 (최대 리스크)** | **우천취소·연기·더블헤더** — KBO 일정 잦은 변동. → E4 2단 주기 + `game_id` upsert로 상태(예정/취소/연기) 덮어쓰기. 우천취소 → 플랜B 유도 연결. |
| 동시성/async | 크롤러 cron 중복 실행 방지(flock/lock). |
| 데이터 | `game_id` 유니크(dedup), 맛집 `(stadium, source_url)` 유니크, 구장 FK 인덱스, upsert 기반 reconcile. |
| 계약/경계 | RLS owner-scoped(E1/E3), 키 격리(E10), 앱은 콘텐츠 쓰기 불가. |
| 리소스 | 셀프호스팅이라 티어 한도·잠자기 없음. 크롤 매너(1일1회 + 경기일 1분, UA·딜레이), 이미지 = Supabase storage. **노출**: 도메인+리버스프록시+TLS로 폰에서 닿게(기존 맥미니 도커 스택 패턴 재사용). |
| 실패 모드 | 크롤 실패 graceful + 텔레그램 알림(E5), HTML 변경 깨짐 모니터링. 맥미니 셀프호스팅 → 백업(pg_dump) 주기화 + 다운 시 앱은 마지막 캐시 표시. |

**Migration/롤백**: Supabase 스키마는 migration SQL로 버전관리, 시딩은 seed 스크립트 분리. 롤백 = migration down + seed 재적재.

## 7. 테스트 케이스

- **T1 일정 연동** — 응원팀 설정 → 해당 팀 *원정* 경기만 필터되어 뜸 (홈/원정 구분 정확)
- **T2 크롤러 정확성** — 크롤 1회 → games가 KBO 실제 일정과 일치, game_id 중복 0
- **T3 우천 reconcile** — "예정"→"취소" 바뀐 경기, 재크롤 후 DB 상태 갱신
- **T4 GPS 스탬프** — 구장 반경(기본 500m) 안=발급, 밖=거부
- **T5 스탬프 중복방지** — 이미 찍은 구장 재발급 안 됨
- **T6 로그인+동기화** — 스탬프 찍고 재설치/재로그인 → 스탬프 생존
- **T7 RLS** — 유저A가 유저B 스탬프 read/write 불가
- **T8 크롤러 graceful** — 소스 깨도(404) 앱은 기존 데이터 표시 + 텔레그램 알림
- **T9 콘텐츠 표시** — 구장 선택 → 가이드·맛집·플랜B 뜨고 출처 링크 클릭 시 원본 열림
- **T10 경로 애니** — 방문 구장 2개+ → 네이버맵에 경로 선 그려짐
- **T11 테마 컬러** — 응원팀 바꾸면 앱 테마 컬러 전환

🎯 **제품 "됐다" 기준**: **"내가 실제 원정 한 번을 이 앱만 보고 다녀올 수 있으면 v1 done."** (일정 확인 → 구장 가이드 → 맛집 → 비 오면 플랜B → 돌아와 스탬프)

## 8. 구현 순서 / 병렬 lane

```
Lane A — 데이터 기반 (선행, 다른 lane이 의존)
  A0 맥미니 Supabase 셀프호스팅(Docker) + 도메인/리버스프록시/TLS (E1)
  A1 Supabase 스키마 + RLS + migration (E1·E3·E10)
  A2 콘텐츠 시딩: 10구장 가이드·맛집·플랜B (pick_type 태깅) (E8)
  A3 일정 크롤러: 맥미니 Python+cron, 2단 주기(경기일 1분)+upsert+텔레그램 알림 (E4·E5)

Lane B — 앱 코어 (A1 이후)
  B1 Flutter 앱 셋업 + 테마 컬러 시스템 (E9)
  B2 로그인: 이메일 + 카카오 (E2)
  B3 응원팀 설정 → 원정 일정 화면 (T1)
  B4 구장 가이드 / 맛집 / 플랜B 읽기 화면 (T9)

Lane C — 인터랙션·게임화 (B1 이후, 일부 A 의존)
  C1 GPS 스탬프 발급 + 클라우드 동기화 (E7·E3, T4~T6)
  C2 네이버맵 위치 + 경로 애니 (E6, T10)
```

**의존성**: A0 → A1 → (B*, C*) 전부 선행. A3는 A1 후. C2는 B1 후. A2(시딩)는 A1 후 병렬 가능.
