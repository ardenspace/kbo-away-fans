# Spec: KBO 원정팬 — Lane C (GPS 스탬프 + 네이버맵)
version: 1
status: approved

## Overview

기존 KBO 원정팬 Flutter 앱(brownfield)의 남은 기능 슬라이스: PLAN.md Lane C
(task-014~017). 구장 GPS 근접 스탬프 발급 + 도장 애니메이션 + 네이버맵 위치
표시 + 방문 구장 경로 애니메이션. 요구사항 상류는 docs/EXECUTION-PLAN.md의
E6·E7·E3와 T4~T6·T10이며, 프로젝트 전체 재기획은 하지 않는다 (사용자 지시:
"PLAN.md의 Lane C… 프로젝트 전체를 재기획하지 말 것").

## Goals

- task-014: GPS 스탬프 코어 — 근접 발급 + 중복방지 + 클라우드 동기화 (E7·E3, T4~T6)
- task-015: 스탬프 도장 애니메이션 (E9)
- task-016: 네이버맵 연동 + 위치 표시 (E6)
- task-017: 지도 경로 애니메이션 — 방문 구장 잇기 (E6, T10)

## Non-Goals

- 프로젝트 전체 재기획 — Lane A·B는 완료·동결, Lane C 외 어떤 task도 다루지 않음
- GPS 스푸핑 대응 — E7: "스푸핑은 dogfood라 무시". 이 면제는 클라이언트 우회
  insert(PostgREST 직접 호출로 임의 stadium_id/lat/lng 삽입)까지 포함하는 넓은
  신뢰 경계임을 명시적으로 인지하고 수용한다 (반경 검증은 클라이언트에만 존재)
- 10구장 완주 뱃지 등 심화 게임화 — EXECUTION-PLAN §4 V1.5/V2 보류 항목
- 백그라운드 위치 추적·지오펜싱·구장 도착 푸시 알림 — 인터뷰에서 수동 버튼 방식 선택으로 배제
- 오프라인 발급 큐(로컬 저장 후 동기화) — 인터뷰에서 "발급은 온라인 필수" 선택
- 자동 발급(앱 열면 감지) — 인터뷰에서 수동 버튼 선택
- 스키마 변경 — stamps/stadiums 테이블·RLS는 task-002 확정 그대로. 단 **시드
  데이터 변경(잠실 LG 행 추가)은 이 슬라이스에 포함** (스키마 아님)
- 크롤러 변경 — 잠실 LG 행 추가 시에도 크롤러 코드·별칭표는 건드리지 않음
- 지도 마커 탭 인터랙션(구장 가이드 딥링크 등) — 마커는 표시 전용
- 접근성 특수 대응(reduced-motion 대체 애니, 비촉각 성공 확인 등) — 1인 dogfood 범위 밖
- 위치 정확도 분기(Android 대략적 위치/iOS 정확한 위치 끔 감지·전용 안내) —
  부정확 위치는 자연스럽게 "반경 밖 + 거리 표시"로 흡수
- 발급 성공 애니의 exactly-once 보장 — insert 커밋 후 응답 유실 → 재시도가
  중복 안내로 흡수되어 애니가 안 나올 수 있음을 수용 (dogfood)
- 스탬프 정정·삭제 경로 — stamps RLS에 UPDATE/DELETE 정책이 없어 오배정(예:
  크롤 공백일 잠실 과발급)은 DB 직접 접근으로만 정정 가능함을 수용 (dogfood)

## Company Lens

- 데드라인 없음 (인터뷰 확인). dogfood 프로젝트 — 완성도 기준은 EXECUTION-PLAN §2
  "과엔지니어링 금지, '내가 쓸 만큼'이 완성도 기준".
- 비용 (인터뷰 확정 — 사용자: "지금은 MVP 개발이니까 [무료 한도 + 한도 설정]으로
  하고 나중에 유료여도 괜찮게끔"): NCP Maps는 종량제이나 대표계정에 API별 월
  무료 이용량이 자동 제공되고(웹 조사: 최소 3,000건~최대 1억 건, Web Dynamic Map
  1,000만 건 사례; Mobile Dynamic Map 과금 단위 = 지도 뷰 onCreate 1건), 콘솔
  **한도 설정으로 무료 이용량 범위 내 사용을 강제 제한**할 수 있다. MVP는 한도
  설정을 ops 절차에 포함해 과금을 구조적으로 차단하고, 향후 유료 전환은 한도만
  풀면 되므로 코드 변경 없이 수용 가능하다.
- **NCP 계정이 아직 없음** (인터뷰 확인) — Client ID 발급(가입→Maps 앱 등록→
  Mobile Dynamic Map 활성화→번들ID/패키지명 등록→한도 설정)은 사람이 수행하는
  외부 블로커. 실행 계획은 task-014·015(키 불필요)를 먼저 배치하고, task-016·017
  시점에 Client ID가 없으면 external-blocker HALT로 사람에게 넘긴다. R9·R10의
  수동(실기기) 확인과 R12의 실키 주입은 키 공급 전까지 보류 항목이다.
- MVP = task-014~017 그 자체. 이 슬라이스가 끝나면 V1 "됐다" 기준(§7: 원정 한 번을
  이 앱만 보고 다녀오기)의 마지막 조각(돌아와 스탬프)이 채워진다.

## User Lens

- 유저 = 도시여행형 원정 KBO 팬 본인 (EXECUTION-PLAN §2, 1인 dogfood).
- 핵심 순간 = 구장 앞에서 "도장 찍기" 버튼을 누르면 스탬프북 칸에 도장이 쾅
  찍히는 순간 (인터뷰: 수동 버튼 선택 — "도장 '찍는' 의식이 게임 컨셉과 맞음").
- 발급 가능 여부 = 순수 GPS 근접만: 경기 여부·내 팀 원정 여부와 무관, 구장
  `stamp_radius_m`(기본 500m) 반경 내면 발급 (E7 문언 그대로, 인터뷰 확인).
- **스탬프북 칸 = 팀 단위 10칸** (인터뷰 확정): 잠실은 두산 칸·LG 칸 2칸으로
  분리(시드에 잠실 LG 행 추가 → stadiums 10행), 나머지 8구장은 구장=팀 1:1.
- **잠실 칸 배정 규칙** (인터뷰, 사용자 원문: "그날 잠실 경기가 두산이면 두산에
  찍어주고 엘지면 엘지에 찍어주고 만약 둘이 붙는 거다 그럼 둘 다 찍어주고" +
  무경기 날은 "둘 다 찍어줌! 그러니까 애니메이션 효과가 연달아 나와야 함!"):
  당일 잠실 경기에 등장하는 팀 중 {두산, LG}에 속하는 팀의 칸에 발급.
  두산-LG 맞대결이면 둘 다, 잠실 무경기 날도 둘 다. 발급 *가능 여부*는 여전히
  근접만이고, 경기 정보는 **칸 배정에만** 쓰인다. 이미 찍힌 칸은 건너뛰고
  새 칸만 발급하며, 대상 칸이 전부 기발급이면 중복 안내.
  - "당일" = **KST 달력일** 기준 (games.scheduled_at은 UTC 저장 — DECISIONS
    task-005의 KST→UTC 변환을 역으로 적용해 날짜 비교; KBO는 KST 리그).
  - 경기 status는 배정에 **무관** (취소·연기 경기도 "그날 잠실 경기"로 취급 —
    "그날 잠실 경기가 두산이면" 문언의 가장 단순한 결정적 해석).
  - 등장 팀 ∩ {두산, LG}가 공집합(이론상 중립 경기)이면 무경기와 동일 취급(둘 다).
  - games 데이터 공백(크롤러 미수집)은 무경기와 구분하지 않는다 — 과발급 가능성을
    dogfood로 수용 (Non-Goals의 정정 경로 부재 항목 참조).
- 중복 = 칸(팀)당 1회 영구 (DB `UNIQUE(user_id, stadium_id)` 그대로 — 잠실
  두 칸은 별도 stadium 행이므로 제약 성립).
- 실패 시 = 거리 피드백형 (인터뷰 선택): 반경 밖이면 가장 가까운 구장명과 남은
  거리("구장까지 N.Nkm", km 소수 1자리), 권한 거부면 설정 앱 유도, 위치 서비스
  꺼짐/네트워크·서버 오류면 원인 안내 + 재시도 가능.
- 진입: 스탬프북은 기존 홈 "내 스탬프" 버튼(/stamp), 지도는 홈에 "원정 지도"
  버튼 신설(/map) (인터뷰 선택).

## Engineer Lens

**보안 (최우선)**
- 위치 데이터는 민감정보: 위치 사용은 **포그라운드 온리** — (a) 발급 순간 1회
  조회, (b) 지도 화면 표시용. 백그라운드 위치 권한은 요청하지 않는다.
  발급 시 lat/lng가 stamps에 저장되며, stamps는 owner-scoped RLS 적용
  완료(007_rls.sql, T7) — 본인 row만 읽기/쓰기 가능. 앱은 anon key만 사용 (E10).
- NCP Maps Client ID는 기존 앱 시크릿 컨벤션(main.dart의 `--dart-define`)을
  따라 주입하고 저장소에 커밋하지 않는다. 단 dart-define은 저장소 위생 조치일 뿐
  최종 바이너리에서 추출 가능하므로, 실질 통제는 NCP 콘솔의 번들ID/패키지명
  등록 + 한도 설정이다. 발급·설정·한도 설정 절차는 task-009의
  ops-kakao-setup.md 패턴대로 ops 문서로 분리 (DECISIONS "시크릿 격리" 준수).
  ops 문서에는 계정 개설 시 콘솔에서 Mobile Dynamic Map의 실제 무료 이용량
  수치와 한도 설정 적용을 확인하는 단계를 포함한다.
  **Client ID 미주입 시 앱은 crash하지 않는다**: 스탬프 기능(task-014·015)은
  무영향, 지도 화면만 안내 표시로 degrade.

**잠실 칸 배정 구현 제약**
- 당일 잠실 경기 조회는 기존 games 테이블 읽기만 사용 (크롤러 무변경). 크롤러는
  잠실 경기를 기존 잠실 행(두산 소속)으로 upsert하므로, 칸 배정은 "해당 날짜 +
  기존 잠실 행의 경기들에 등장하는 홈·원정 팀 ∩ {두산, LG}"로 판정한다.
- 잠실 LG 행은 **시드만 추가**하며, name은 크롤러 구장 별칭표(짧은명→공식명
  "잠실야구장")와 충돌하지 않는 고유값으로 한다 (예: "잠실야구장 (LG)") —
  크롤러가 LG 행으로 upsert하는 일이 없어야 한다.

**실패 모드** (인터뷰: 거리 피드백형 — "재시도 가능" = "도장 찍기" 버튼이
재활성화되어 다시 누르면 발급 플로우가 재호출되는 상태)
- 위치 권한 거부 → 설정 앱 유도 버튼과 함께 안내, crash 없음.
- 위치 서비스 꺼짐 → 발급하지 않고 원인 안내.
- GPS fix 미획득(실내 등) → 위치 조회에 타임아웃(수 초 수준, plan에서 확정)을
  두고 초과 시 실패 안내 + 버튼 재활성 — 로딩 무한 고착 금지.
- 네트워크·서버 오류(오프라인, 백엔드 다운, 5xx, 타임아웃 — 구분하지 않고 동일
  취급) → 발급하지 않고 "연결할 수 없음" 안내 + 재시도 가능. 발급은 온라인 필수
  (스탬프 진실원천 = 클라우드, E3).
- 반경 밖 → 가장 가까운 구장명 + 남은 거리 표시. 잠실 두 행처럼 동일 좌표
  동률이면 기존 행 name("잠실야구장")을 표시 — LG 행의 내부 합성 name은 UI에
  노출하지 않는다 (스탬프북 칸 라벨은 팀명 기준).
- 중복 발급 시도 → 클라이언트 선판정하되, 경합 시 UNIQUE 에러도 동일 UX로 흡수.
- 복수 칸 발급의 부분 실패(첫 insert 성공, 이후 실패) → 성공한 칸의 애니는
  재생하고 실패 안내 + 재시도 가능; 재시도는 기발급 칸 건너뛰기로 자연 수렴.
- 발급 진행 중(GPS fix 대기~insert 완료) 버튼 비활성 + 로딩 표시 — 연타로 인한
  중복 요청 방지.
- 스탬프북·지도 화면의 데이터 조회(stamps/stadiums) 실패 → 빈 상태("0/10",
  빈 지도)로 렌더하지 않고 로딩/오류 상태 + 재시도를 표시.

**테스트 전략** (사용자 지시: "Flutter 단위/위젯 테스트, GPS는 mocking 필요")
- 위치 조회를 riverpod provider로 추상화하고, 스탬프 저장·구장·경기 조회는
  데이터 계층(repository) 뒤에 두어 테스트에서 **위치 mock + 데이터 계층 fake**
  주입 — 근접 판정·거리 계산·잠실 칸 배정·중복 방지·실패 분기를 실기기 없이
  단위/위젯 테스트.
- 네이버맵 위젯은 네이티브 SDK라 위젯 테스트 불가 영역 — 마커 목록·경로 좌표
  시퀀스·애니 트리거 조건 등 지도 로직을 순수 Dart로 분리해 단위 테스트하고,
  지도 렌더링 자체는 실기기 수동 확인으로 넘긴다 (dogfood 기준, Client ID 공급
  후 수행).
- 가장 확신을 주는 한 테스트: "반경 안 mock 위치로 발급 → fake 저장 계층에 기록 +
  반경 밖 mock 위치로 거부" (T4).

**과엔지니어링 경계**
- 스푸핑 대응·지오펜싱·오프라인 큐·백그라운드 추적·정확도 분기 없음 (Non-Goals).
- 거리 계산은 하버사인 수준이면 충분 — 측지 정밀도 불요.

**플랫폼·컨벤션**
- iOS·Android 둘 다 동등 지원 (인터뷰 확인): 위치 권한(Info.plist /
  AndroidManifest)과 네이버맵 네이티브 설정을 양쪽 모두 구성.
- 기존 컨벤션 준수 (사용자 지시): feature 폴더 구조(app/lib/features/stamp,
  features/map — 디렉토리 기존재), riverpod codegen(@Riverpod + build_runner),
  go_router 라우트 추가 방식, 팀 테마 시스템(shared/theme).
- 지도 SDK = `flutter_naver_map` (E6 lock-in). 위치 플러그인은 mock 주입 가능한
  추상화 뒤에 두는 것이 요건이며 구체 패키지 선택은 plan에 위임.

## Designer Lens

  실패 + 재시도" 상태를 표시한다 — 빈 "0/10"으로 렌더하지 않음.

  프리미티브로 구현 — 외부 에셋(Lottie/Rive) 불필요. **복수 칸 동시 발급(잠실

  사용자 원문: "애니메이션 효과가 연달아 나와야 함").
- 지도 화면 = 홈 "원정 지도" 버튼 → /map (인터뷰 선택): 네이버맵에 구장 마커
